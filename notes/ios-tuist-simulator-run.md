# Run StarterApp (Tuist) on the iOS Simulator

This app lives under `ios/StarterApp`. The Xcode project is **generated by Tuist**, not checked into git (see `ios/StarterApp/.gitignore`).

## Quick start (recommended)

A script handles all the steps below automatically:

```bash
# First time / after cloning (runs tuist install + generate first)
make ios-sim-regen

# Normal day-to-day build + launch
make ios-sim

# Build + launch + stream console logs
make ios-sim-logs

# Target a specific simulator by UDID
./ios/StarterApp/run-sim.sh --udid <UDID>
```

The script (`ios/StarterApp/run-sim.sh`) auto-picks the newest available iPhone simulator, builds with `-quiet`, then boots → installs → launches in one go. The manual steps below are kept for reference/troubleshooting.

---

## Prerequisites

1. **Xcode** with at least one **iOS Simulator runtime** installed (Xcode → Settings → Platforms).
2. **Tuist** installed ([tuist.dev](https://docs.tuist.dev)) so you can regenerate the workspace when manifests change, for example:
   ```bash
   cd ios/StarterApp
   tuist install
   tuist generate --no-open
   ```
3. **Local config**: copy `Config.example.xcconfig` to `Config-Debug.xcconfig` (and edit). The app target reads `PRODUCT_BUNDLE_IDENTIFIER` from this file. Without it, builds can succeed but the built app may omit `CFBundleIdentifier`, and CLI install/launch breaks. Set at least:
   - `PRODUCT_BUNDLE_IDENTIFIER` (e.g. `com.example.StarterApp`)
   - Your backend/Supabase URLs and keys as documented in the example file

## Generate or refresh the workspace

From the Tuist root:

```bash
cd ios/StarterApp
tuist install    # resolves Tuist/Package.swift (Supabase, PostHog, etc.)
tuist generate --no-open
```

Open `StarterApp.xcworkspace` in Xcode and press Run, or use the command line below.

## List schemes

```bash
cd ios/StarterApp
xcodebuild -list -workspace StarterApp.xcworkspace
```

Typical scheme for the app: **StarterApp**.

## Pick a simulator (device + iOS version)

Apple often installs **multiple runtimes** with the same device name (e.g. several “iPhone 16”). Using only `name=iPhone 16` can fail with “no available devices matched” because **OS=`latest` is ambiguous**.

**Option A — list UDIDs and choose one**

```bash
xcrun simctl list devices available
```

Find a line like `iPhone 16 (...)` with the **OS** you want and copy the **UDID** in parentheses.

**Option B — disambiguate in `xcodebuild`**

Use an explicit OS (or the UDID):

```bash
# By name + OS
-destination 'platform=iOS Simulator,name=iPhone 16,OS=18.5'

# By UDID (most precise)
-destination 'platform=iOS Simulator,id=YOUR-UDID-HERE'
```

**Generic destination** (any Simulator; good for CI when you do not care which model):

```bash
-destination 'generic/platform=iOS Simulator'
```

## Build for Simulator

Example: iPhone 16, iOS 18.5, isolated DerivedData next to the project:

```bash
cd ios/StarterApp
xcodebuild build \
  -workspace StarterApp.xcworkspace \
  -scheme StarterApp \
  -configuration Debug \
  -destination 'platform=iOS Simulator,name=iPhone 16,OS=18.5' \
  -derivedDataPath ./DerivedDataRun
```

The `.app` path after a successful build:

`DerivedDataRun/Build/Products/Debug-iphonesimulator/StarterApp.app`

## Boot, install, and launch (CLI)

Replace `YOUR_UDID` and the `.app` path if yours differ.

```bash
UDID=YOUR_UDID
APP="$(pwd)/DerivedDataRun/Build/Products/Debug-iphonesimulator/StarterApp.app"
BUNDLE_ID=com.example.StarterApp   # must match PRODUCT_BUNDLE_IDENTIFIER in Config-Debug.xcconfig

xcrun simctl boot "$UDID" 2>/dev/null || true
open -a Simulator --args -CurrentDeviceUDID "$UDID"
xcrun simctl install "$UDID" "$APP"
xcrun simctl launch "$UDID" "$BUNDLE_ID"
```

To stream logs in the terminal (blocks until the app exits):

```bash
xcrun simctl launch --console-pty "$UDID" "$BUNDLE_ID"
```

## Different simulators / iOS versions — quick reference

| Goal | What to change |
|------|----------------|
| Different **phone model** | Pick another device in `simctl list` (e.g. iPhone 17 Pro) and use its **name+OS** or **UDID** in `-destination` |
| Different **iOS** for same model | Same device name, different **OS** in the list; use `name=...,OS=x.y` or **UDID** |
| Avoid “ambiguous destination” | Prefer **UDID** or always set **`OS=`** |
| Regenerate after `Project.swift` / `Tuist/Package.swift` change | `tuist install && tuist generate --no-open` |

## Troubleshooting

- **`xcodebuild` / `simctl` errors in sandboxed or restricted environments**: run the same commands in Terminal locally; Simulator needs CoreSimulator services.
- **Workspace missing**: run `tuist install && tuist generate` from `ios/StarterApp`.
- **Install works but launch fails**: confirm `PRODUCT_BUNDLE_IDENTIFIER` in `Config-Debug.xcconfig` and that `plutil -p …/StarterApp.app/Info.plist` shows `CFBundleIdentifier`.
