### `backend/tests/integration/test_migration_drift.py` (new file)
- Integration test (`@pytest.mark.integration`) that:
  1. Drops all user tables plus `alembic_version` on `postgres_test`
  2. Runs `alembic upgrade head` in a subprocess (keeps Alembic's async env.py off pytest's event loop)
  3. Opens a fresh async engine and uses `alembic.autogenerate.compare_metadata` via `conn.run_sync` to diff the live schema against `Base.metadata`
  4. Fails the test if any non-benign diffs are found
  5. Filters `modify_server_default` diffs (benign dialect representation differences for Boolean server defaults)
  6. Teardown drops user tables but keeps `alembic_version` at head, so subsequent `alembic upgrade head` invocations (including the Makefile's pre-test step) are idempotent

## Key discovery: pre-existing drift in migration 001

The drift guard immediately caught real drift: migration 001 used project-local index names that differ from SQLAlchemy's autogenerated names. This is genuine drift that would have caused silent schema differences between CI (using `create_all`) and production (using Alembic). Migration 003 fixes this.

## Code-review fix pass (2026-07-02)

### Finding 1 — drift filter too broad (`backend/tests/integration/test_migration_drift.py`)

`_is_benign_server_default_diff` previously returned `True` for any diff where
`diff[0] == "modify_server_default"`, which would silently hide future real drift
on any new column with a server default.

Confirmed from `backend/app/db/models.py` and the migrations:
- `users.is_active` — `server_default='true'` (migration 001)
- `profiles.is_pro` — `server_default='false'` (migration 002)

These are the only Boolean columns with server defaults. The filter was narrowed to
check `(diff[2], diff[3])` (table name, column name) against the explicit set
`{("users", "is_active"), ("profiles", "is_pro")}`. Any other `modify_server_default`
diff on a different column now returns `False` (test fails).

### Finding 2 — misleading docstring (`backend/tests/api/test_auth_routes.py`)

`test_register_rejects_password_too_short` docstring said "7-byte password" but the
test sends `"short"` (5 chars). Corrected to "5-char password".

### Finding 3 — weak assertion (`backend/tests/api/test_auth_routes.py`)

`test_register_accepts_password_at_max_length` asserted `status_code != 422`,
which would pass on a 500 and mask real failures. The `/register` endpoint returns
`201` on success and `409` on duplicate email; when the DB is unavailable in the
unit suite it returns `500`. Tightened to:

```python
assert response.status_code in (201, 409, 500)
```

## Concerns

1. **Migration 003 renames production indexes.** This is a destructive rename for existing installations. The rename is safe (drop + create with `IF EXISTS` guarding via Alembic's transactional DDL), but deployers with existing data should be aware. The rename is necessary to keep the ORM and Alembic in sync.

2. **Teardown ordering assumption.** The drift test teardown keeps `alembic_version` at head rather than dropping everything. This assumes the Makefile's `alembic upgrade head` runs before pytest each time (which it does), and that the alembic_version table is not cleaned up by `Base.metadata.drop_all` (correct — alembic_version is not in Base.metadata). Verified stable across two consecutive invocations.

3. **`alembic downgrade base` is broken for existing `create_all` DBs.** The downgrade path in migration 001 tries to drop named indexes that only exist if migrations were used (not `create_all`). This is a pre-existing issue not introduced by this PR. The drift test avoids downgrade entirely, using `drop_all` + `upgrade head` instead.
