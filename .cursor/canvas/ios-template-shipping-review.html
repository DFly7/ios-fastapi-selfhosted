Executive summary

This is a strong production-oriented starter: Tuist as the iOS source of truth, xcconfig-driven secrets, FastAPI with JWKS verification and real RLS migrations, integration tests against local Supabase, and a documented TestFlight path. The main friction is onboarding surface area (several docs, env name splits, gitignored configs), template renaming (many “StarterApp” / com.example touchpoints, no single rename script yet), and a few doc/CI mismatches plus at least one workflow bug called out below.

Template usability

What works exceptionally well

  • One command dev story — make dev and documented Supabase + backend + tunnel flows collapse weeks of glue.
  • Reproducible toolchains — mise pins Tuist, Supabase CLI, SwiftLint, etc.
  • CI honesty (mostly) — Backend unit tests, model sync check, iOS Tuist generate + unit tests mirror local targets.
  • Security-conscious DB layer — RLS, trigger-only profile creation, hardened is_pro policies, waitlist rate limits in SQL.

Friction for a new cloner

  • Generated Xcode projects are gitignored — must run make ios-gen (or Tuist) before Xcode is useful.
  • Local xcconfigs and backend .env are gitignored — copy from examples; same pattern as CI synthesis.
  • Ruby / Bundler for pretty test output — make ios-test and make ios-build use bundle exec xcpretty from ios/StarterApp; root README does not front-load bundle install.
  • Multiple “how to run” narratives — README.md, local-setup.md, backend/README.md, manual multi-terminal flows.
  • Env naming mental map — backend SUPABASE_PUBLIC_ANON_KEY vs iOS / GH secrets as SUPABASE_ANON_KEY.
  • Simulator selection — Makefile heuristic can pick a simulator that disagrees with the selected Xcode SDK (CI documents this class of risk).

Path to App Store

Strengths

  • Privacy manifest present with tracking declaration, collected data types, UserDefaults reason (PostHog-related).
  • ATS / local networking configured for the local-dev story.
  • Capabilities — Sign in with Apple and In-App Purchase entitlements wired in Tuist.
  • Fastlane + Match — ios/StarterApp/fastlane/; Makefile targets setup-dist, beta, release; distribute.yml on version tags for TestFlight.
  • Export compliance — non-exempt encryption flag set with explanation in project definition.
  • Release xcconfig — dSYM-friendly settings; placeholders for production URLs.

Gaps / watch items

  • UI tests — still default XCTest scaffold; not exercised in iOS CI (only StarterAppTests), despite README implying broader test CI in one line — align docs or add UITests to the workflow.
  • Bundle ID coherence — app id comes from xcconfig; test bundle IDs in Project.swift may drift if only xcconfig is edited.
  • Backend deploy — Docker build / optional GHCR push exists; no included workflow deploys FastAPI to a host (template stops at image).

Skeleton code & conventions

Verdict: good, opinionated, teachable

  • Clear MV-oriented layout — Views/, ViewModels/, Services/, Models/, Utilities/.
  • Modern Swift — Swift Testing for units; concurrency flags and MainActor defaults in Tuist.
  • API contract discipline — Pydantic schemas + scripts/sync_models.py → GeneratedModels.swift (SwiftLint excludes generated file).
  • Backend layering — api/, core/, middleware/, services/, repositories/, schemas/ is easy to extend.
  • Tuist discipline — matches repo rules: edit Project.swift and Tuist/Package.swift, not generated pbxproj.

Small inconsistencies to tighten

  • Profile API vs schema — ProfileOut includes is_pro but route select may omit DB column until aligned — responses won’t reflect subscription state from DB.
  • Localization flags in project without a committed .xcstrings yet — fine for a skeleton, worth one sample string catalog when you want emphasis on i18n.

Developer experience

  • Makefile is the spine: validate, lint, backend-test, ios-test, ios-build, check-models, distribution helpers.
  • Secrets workflow documented in .github/CI_GITHUB_SECRETS.md.
  • Production ops caveats (know them early): in-process rate limits unless Redis storage configured; optional metrics endpoint needs network protection; webhook may warn-and-accept if secret unset; request body logging default — tune for production.

Agent X (Cursor & autonomous agents)

What agents get today

  • Always-on monorepo.mdc — root commands, Tuist edit surfaces, docs/ policy, Supabase MCP caveats.
  • Excellent AGENTS.md — stack map, testing tables, MCP hosted vs local, verification commands, internal vs published docs.
  • Large skill library (~75 topics) — high value when matched to task; risk of overload unless runtime curates relevance.
  • mcp.json.example — cloud + local Supabase MCP URLs; gitignored real config.

Gaps for agent ergonomics

  • No root CLAUDE.md / GEMINI.md while some skills still reference them.
  • Single rule file — optional path-scoped rules for backend/ vs ios/ would reduce cross-layer mistakes.
  • No “after clone” checklist in AGENTS.md for renaming bundle, URL scheme, Supabase project ref, MCP — agents benefit from one canonical list.
  • Using-superpowers skill assumes non-Cursor tool names; Cursor users rely on AGENTS.md + rules as the real contract.

Confirmed issues & doc drift

Medium

  • README claims UI tests “running in CI” — ios-ci.yml only runs StarterAppTests.
  • CI comment vs Project.swift: signing disabled via CLI for simulator; verify comments stay accurate.
  • Optional GHCR image push gated on PUBLISH_BACKEND_IMAGE — README table can state that explicitly.

Next steps — rename / fork automation

Your idea: one script that renames everything “StarterApp-related” to a user-supplied app name. A robust script should cover at least:

  • Tuist — target names, scheme, module imports in tests, Project.swift paths, product name.
  • Directories — ios/StarterApp/StarterApp → new app folder; test targets folders.
  • Bundle IDs & URL scheme — Config.example.xcconfig, entitlements, Supabase redirect URLs, APIConfig / Info.plist keys if hardcoded.
  • Swift — @testable import, app struct name (StarterAppApp → MyAppApp), any string literals.
  • Repo metadata — README title, AGENTS.md paths, workflow -only-testing: strings, Fastlane app identifiers.
  • Supabase / backend — cosmetic references only unless you also template Supabase project id.
  • Dry-run mode and idempotency — show diff before write; optional backup branch.

Implementation sketch: small Python or shell driver + explicit allowlist of glob patterns to avoid rewriting .git, lockfiles, and binaries; run make check-models and make ios-gen after.
